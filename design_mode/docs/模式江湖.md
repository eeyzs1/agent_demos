## 🧠 谋士门 — 推理增强类

> **门派心法**：三思而后行。谋士们擅长让 LLM (Large Language Model，大语言模型) 在开口前先把问题想透。
> **完整武学**：[01_推理增强类.md](01_推理增强类.md)

### 1.1 CoT (Chain of Thought，思维链) — "步步为营"

> **原理**：在提示词中加入"Let's think step by step"引导语，让 LLM 显式展开中间推理步骤后再给结论，通过分步推理提升复杂问题的准确率。

| 属性 | 内容 |
|------|------|
| **名号** | 步步为营 |
| **一句话记忆** | "把心算变成草稿纸" |
| **生动类比** | 就像数学老师要求写解题步骤——不写步骤直接给答案容易出错，一步一步写下来准确率飙升 |
| **招式特点** | 在 prompt 里加一句"Let's think step by step"，LLM 就会逐步推理 |
| **最佳战场** | 数学计算、逻辑推理、多步决策 |
| **致命弱点** | 推理链太长时会"走神"，中间步骤出错会一路错到底 |

---

### 1.2 ToT (Tree of Thoughts，思维树) — "分身术"

> **原理**：将推理过程组织为树结构，同时探索多条分支路径，对每条路径评估前景并剪枝，最终回溯选出最优解，突破 CoT 单链推理的局限。

| 属性 | 内容 |
|------|------|
| **名号** | 分身术 |
| **一句话记忆** | "不撞南墙不回头？不，我派分身探每条路" |
| **生动类比** | 就像下棋时不是只想一步，而是同时推演"走车、走马、走炮"三条路，每条路再往前想几步，最后选最优的 |
| **招式特点** | 生成多个推理分支，评估每个分支的前景，剪枝保留有希望的路径 |
| **最佳战场** | 创意写作、填字游戏、需要探索的复杂推理 |
| **致命弱点** | 计算/调用成本是 CoT 的 N 倍（N=分支数×深度） |

---

### 1.3 GoT (Graph of Thoughts，思维图) — "思维织网"

> **原理**：将推理过程建模为图结构（而非线性链或树），节点之间可合并、循环、回溯，支持多线索交叉验证和反复修正，适合需要综合多方面信息的复杂推理。

| 属性 | 内容 |
|------|------|
| **名号** | 思维织网 |
| **一句话记忆** | "线索不是链条，是网" |
| **生动类比** | 侦探把线索连成关系网而非线性推理，多条线索互相印证才能破案 |
| **招式特点** | 把推理过程建模成图结构，节点可合并、循环、回溯，打破线性链条 |
| **最佳战场** | 复杂问题综合分析、多线索交叉验证、需要反复修正的推理 |
| **致命弱点** | 图结构管理复杂，"网"织大了容易理不清头绪 |

---

### 1.4 Self-Consistency — "多重影分身"

> **原理**：对同一问题采样多条独立推理路径，通过多数投票选出出现频率最高的答案，利用"多路径一致性"降低单次推理的随机误差。

| 属性 | 内容 |
|------|------|
| **名号** | 多重影分身 |
| **一句话记忆** | "找多个人独立做题，答案一致才放心" |
| **生动类比** | 考试后找5个同学对答案，多数一致的那个才靠谱 |
| **招式特点** | 对同一问题采样多条推理路径，投票选出出现最多的答案 |
| **最佳战场** | 数学题、有唯一正确答案的推理任务 |
| **致命弱点** | 烧钱（要跑 N 次），对开放性问题投票效果差 |

---

### 1.5 PoT (Program of Thoughts，程序化思维) — "代码会算"

> **原理**：让 LLM 生成可执行的程序代码来求解数学/逻辑问题，而非用自然语言推理，通过代码的精确计算能力避免 LLM 的数值运算错误。

| 属性 | 内容 |
|------|------|
| **名号** | 代码会算 |
| **一句话记忆** | "心算不如编程算" |
| **生动类比** | 用计算器代替心算，让LLM写代码来算数学 |
| **招式特点** | 让 LLM 生成程序代码来求解，再执行代码得到精确结果 |
| **最佳战场** | 数值计算、统计分析、需要精确结果的数学题 |
| **致命弱点** | 代码写错了照样算错，且需要代码执行环境 |

---

### 1.6 Step-Back — "退一步海阔天空"

> **原理**：先引导模型从具体问题中抽象出底层原理或概念，再基于原理回到具体问题作答，通过"先见森林再见树木"降低直接推理的难度。

| 属性 | 内容 |
|------|------|
| **名号** | 退一步海阔天空 |
| **一句话记忆** | "先看森林，再看树木" |
| **生动类比** | 迷路时先看地图大局，再找具体路线 |
| **招式特点** | 先引导模型抽象出问题背后的原理/概念，再回到具体问题作答 |
| **最佳战场** | 科学问题、多步推理、需要调用底层知识的复杂题 |
| **致命弱点** | 简单问题也"退一步"反而绕远路，杀鸡用牛刀 |

---

### 1.7 Least-to-Most — "剥洋葱"

> **原理**：先将复杂问题分解为由易到难的子问题序列，依次解决，前一步的答案作为后一步的输入，通过逐步降难实现复杂问题求解。

| 属性 | 内容 |
|------|------|
| **名号** | 剥洋葱 |
| **一句话记忆** | "从最外层开始，一层层剥" |
| **生动类比** | 拆快递：先拆大箱，再拆小盒，最后拿东西 |
| **招式特点** | 先把复杂问题分解成由易到难的子问题序列，依次解决，前一步结果喂给后一步 |
| **最佳战场** | 多步推理、复杂任务分解、教学场景 |
| **致命弱点** | 分解错了全盘皆输，依赖问题本身可分解 |

---

### 1.8 Analog Prompting — "以史为鉴"

> **原理**：让模型自动生成与当前问题类似的类比案例，再借类比推理迁移到新问题，利用"相似问题有相似解法"的迁移学习思想。

| 属性 | 内容 |
|------|------|
| **名号** | 以史为鉴 |
| **一句话记忆** | "历史不会重复，但会押韵" |
| **生动类比** | 用"上次类似问题怎么解决的"来启发新问题 |
| **招式特点** | 让模型自动生成相关类比案例，再借类比推理解决新问题 |
| **最佳战场** | 创新问题、跨领域迁移、缺乏直接示例的场景 |
| **致命弱点** | 类比不当会"东施效颦"，强行套用反而误导 |

---

### 1.9 SoT — "先骨架后血肉"

> **原理**：先让模型生成答案的骨架提纲，再将各要点并行展开填充，通过"先框架后内容"的两阶段策略大幅加速长文本生成。

| 属性 | 内容 |
|------|------|
| **名号** | 先骨架后血肉 |
| **一句话记忆** | "先写提纲，再填内容" |
| **生动类比** | 建房子先搭框架再砌墙 |
| **招式特点** | 先让模型生成答案的骨架（提纲/要点），再并行填充各部分内容 |
| **最佳战场** | 长文生成、报告撰写、需要加速的长输出任务 |
| **致命弱点** | 骨架没搭好，血肉再丰满也是歪的 |

---

### 1.10 RE2 (Re-Reading，重复阅读) — "重要的事说三遍"

> **原理**：让模型把输入问题重述一遍再回答，通过二次阅读增强对题意的理解，减少因误读题目导致的回答偏差。

| 属性 | 内容 |
|------|------|
| **名号** | 重要的事说三遍 |
| **一句话记忆** | "读题读两遍，理解更全面" |
| **生动类比** | 考试时把题目读两遍再答题 |
| **招式特点** | 让模型把问题重述一遍再回答，强化对题意的理解 |
| **最佳战场** | 题意复杂、容易理解偏差的任务 |
| **致命弱点** | 多花一遍的 token，简单题上纯属浪费 |

---

### 1.11 S2A (System 2 Attention，系统2注意力) — "深思熟虑"

> **原理**：先将原始问题重述为去偏见、去情绪的版本，再基于重述后的问题作答，模拟卡尼曼"系统2"的慢思考来过滤直觉偏见。

| 属性 | 内容 |
|------|------|
| **名号** | 深思熟虑 |
| **一句话记忆** | "先过滤偏见，再思考问题" |
| **生动类比** | 像冷静下来后再做决定，不被第一反应带偏 |
| **招式特点** | 先把问题"重述"为去偏见、去情绪的版本，再基于重述后的问题作答 |
| **最佳战场** | 含偏见/诱导性表述的问题、需要客观判断的任务 |
| **致命弱点** | 重述可能丢失原问题关键信息，"过滤过头" |

---

### 1.12 Emotion Prompting — "情绪鼓励"

> **原理**：在提示词中加入情绪刺激语（如鼓励、重要性强调），利用心理学效应激发模型更积极的推理表现。

| 属性 | 内容 |
|------|------|
| **名号** | 情绪鼓励 |
| **一句话记忆** | "一句'你能行'，性能涨5%" |
| **生动类比** | 像考前老师拍肩膀说"这题你肯定行" |
| **招式特点** | 在 prompt 中加入情绪刺激语（鼓励、重要性强调），提升模型表现 |
| **最佳战场** | 通用任务加成、需要提升模型"积极性"的场景 |
| **致命弱点** | 鼓励过头像 PUA，效果因模型而异，玄学成分偏高 |

---

### 1.13 LATS (Language Agent Tree Search，语言智能体树搜索) — "棋圣"

> **原理**：将蒙特卡洛树搜索、价值评估、反思和行动融为一体，在 Agent 任务中做完整的多步推演，结合环境反馈动态调整搜索方向。

| 属性 | 内容 |
|------|------|
| **名号** | 棋圣 |
| **一句话记忆** | "我不仅想这步，我还想对手会怎么应对" |
| **生动类比** | 围棋大师的蒙特卡洛树搜索，推演+行动+反思一体 |
| **招式特点** | 把蒙特卡洛树搜索、价值评估、反思和行动合一，在Agent任务中做完整推演 |
| **最佳战场** | 复杂决策任务、需要长程规划的Agent场景 |
| **致命弱点** | 计算量极大，下盘"棋"要烧不少 token |

---

### 1.14 Test-Time Compute Scaling — "深思者"

> **原理**：通过 RL+PRM 训练让模型内化深度推理能力，推理时自主生成隐藏思维链，无需外部提示即可进行长时间深度思考（如 o1 模型）。

| 属性 | 内容 |
|------|------|
| **名号** | 深思者 |
| **一句话记忆** | "不用你催我想，我自己会想很久" |
| **生动类比** | 像o1模型，拿到题先沉默思考很久，然后给出惊艳答案 |
| **招式特点** | 推理时自主生成隐藏思维链，通过RL (Reinforcement Learning，强化学习)+PRM (Process Reward Model，过程奖励模型)训练内化深度推理能力，无需外部提示 |
| **最佳战场** | 数学竞赛、编程题、科学推理等需要深度思考的复杂任务 |
| **致命弱点** | 简单任务反而浪费时间和token，思维链不可见难以调试 |

---

### 1.15 Chain-of-Verification (CoV，验证链) — "质检员"

> **原理**：生成初始回答后，自主规划验证问题并独立回答，再将验证结果与原回答比对修正，通过"自我出题验证"闭环降低事实性错误。

| 属性 | 内容 |
|------|------|
| **名号** | 质检员 |
| **一句话记忆** | "写完答案自己出题验证一遍" |
| **生动类比** | 像考试写完卷子后自己出几道验证题，确认答案没问题再交卷 |
| **招式特点** | 生成初始回答→规划验证问题→独立回答验证→比对修正，四步闭环 |
| **最佳战场** | 事实密集型问答、需要高可靠性的知识问答场景 |
| **致命弱点** | 验证问题可能和原问题同源，独立验证不一定能发现系统性错误 |

## ⚔️ 将帅门 — 自主规划与执行类

> **门派心法**：运筹帷幄，决胜千里。将帅们让 Agent 能自主拆解目标、调度资源、执行任务。
> **完整武学**：[02_自主规划与执行类.md](02_自主规划与执行类.md)

### 2.1 AutoGPT/BabyAGI — "自主将军"

> **原理**：给定高层目标后，Agent 自主拆解任务、管理任务队列、循环执行，实现"目标驱动"的全自动任务完成。

| 属性 | 内容 |
|------|------|
| **名号** | 自主将军 |
| **一句话记忆** | "给我一个目标，还你一个结果" |
| **生动类比** | 像给将军一个战略目标，他自己排兵布阵打仗 |
| **招式特点** | 给定高层目标，Agent 自动拆解任务、循环执行、自我管理任务队列 |
| **最佳战场** | 开放式目标达成、长程自动化任务 |
| **致命弱点** | 容易"跑偏"无限循环，token 烧得飞快却未必有结果 |

---

### 2.2 HuggingGPT — "调度员"

> **原理**：LLM 充当调度大脑，解析任务后调度 HuggingFace 上各类专家模型（视觉、语音、语言等）协同完成多模态任务。

| 属性 | 内容 |
|------|------|
| **名号** | 调度员 |
| **一句话记忆** | "我是项目经理，专家们听我调" |
| **生动类比** | 像项目经理调度设计、开发、测试各路专家 |
| **招式特点** | LLM 当大脑解析任务，调度 HuggingFace 上各类专家模型完成感知、推理、生成 |
| **最佳战场** | 多模态任务、需要调用多种专用模型的复杂任务 |
| **致命弱点** | 依赖外部模型可用性，调度链路一断全盘停 |

---

### 2.3 DEPS (Describe, Explain, Plan, Select，描述-解释-规划-选择) — "导演"

> **原理**：通过 Describe（描述状态）→Explain（解释差距）→Plan（规划候选）→Select（选择最优）四阶段，结构化地管理 Agent 的子目标选择。

| 属性 | 内容 |
|------|------|
| **名号** | 导演 |
| **一句话记忆** | "先写剧本，再按剧本拍" |
| **生动类比** | 像导演先写分镜脚本再开拍 |
| **招式特点** | 先生成可执行的"剧本"（任务脚本），再按脚本逐步执行，支持错误恢复 |
| **最佳战场** | 机器人控制、具身智能、需要可解释执行流程的任务 |
| **致命弱点** | 剧本写死了不够灵活，临场变化难应对 |

---

### 2.4 ART (Automatic Reasoning and Tool-use，自动多步推理与工具调用) — "学徒"

> **原理**：从少量示例中学习任务流程并自动选择/调用工具，执行后将新经验存回任务库，实现"看一遍学一遍"的跨任务迁移。

| 属性 | 内容 |
|------|------|
| **名号** | 学徒 |
| **一句话记忆** | "看一遍，学一遍，做一遍" |
| **生动类比** | 像实习生看前辈怎么做，学着做，越做越好 |
| **招式特点** | 从少量示例中学习任务流程，执行后还能把新经验存回任务库 |
| **最佳战场** | 新任务快速上手、可复用流程积累 |
| **致命弱点** | 示例质量决定上限，"师傅带歪了徒弟也歪" |

---

### 2.5 ADaPT (Adaptive Decomposition and Planning for Tasks，自适应分解与规划) — "随机应变"

> **原理**：子任务执行失败时动态重新分解和调整计划，而非整体放弃，通过"此路不通换条路"的适应性策略提升长程任务完成率。

| 属性 | 内容 |
|------|------|
| **名号** | 随机应变 |
| **一句话记忆** | "此路不通？换条路走" |
| **生动类比** | 像导航遇到堵路自动重新规划路线 |
| **招式特点** | 子任务执行失败时，动态调整和重新分解任务，而非整体放弃 |
| **最佳战场** | 不确定环境下的长程任务、容易出错的执行链 |
| **致命弱点** | 反复重试可能陷入"死循环换路"，越改越乱 |

---

### 2.6 Plan-and-Solve — "先计划后执行"

> **原理**：先让模型把任务拆成明确计划，再按计划逐步执行，通过"先规划后执行"两阶段降低直接执行的盲目性。

| 属性 | 内容 |
|------|------|
| **名号** | 先计划后执行 |
| **一句话记忆** | "磨刀不误砍柴工" |
| **生动类比** | 像旅行前先做攻略再出发 |
| **招式特点** | 先让模型把任务拆成明确计划，再按计划逐步执行 |
| **最佳战场** | 复杂多步任务、需要清晰步骤的执行场景 |
| **致命弱点** | 计划赶不上变化，执行中遇到意外就卡壳 |

---

### 2.7 LLM+P (LLM + Planner，大语言模型+经典规划器) — "军师+将军"

> **原理**：LLM 负责将自然语言问题翻译成 PDDL 形式化描述，经典符号规划器求解后再转回自然语言，结合 LLM 的语言能力和规划器的严谨性。

| 属性 | 内容 |
|------|------|
| **名号** | 军师+将军 |
| **一句话记忆** | "谋士出策，将军执行" |
| **生动类比** | LLM当军师出策略，经典规划器当将军执行 |
| **招式特点** | LLM 把自然语言问题翻译成 PDDL (Planning Domain Definition Language，规划领域定义语言)，交给经典符号规划器求解，再把结果转回人话 |
| **最佳战场** | 需要严格可行性的规划任务、传统规划问题 |
| **致命弱点** | 翻译成 PDDL 这步容易出错，且规划器领域受限 |

---

### 2.8 ReWOO (Reasoning Without Observation，无观察推理) — "批量采购"

> **原理**：一次性规划好所有工具调用的依赖关系，批量并行执行后统一求解，通过"推理与观察解耦"大幅减少中间 LLM 调用次数。

| 属性 | 内容 |
|------|------|
| **名号** | 批量采购 |
| **一句话记忆** | "先列清单，一次买齐，别跑八趟" |
| **生动类比** | 像先列购物清单一次性去超市，而不是买一样跑一趟 |
| **招式特点** | 一次性规划好所有工具调用依赖关系，批量并行执行，最后统一求解 |
| **最佳战场** | 多工具调用任务、想省 token 的规划场景 |
| **致命弱点** | 清单列漏了得返工，依赖关系判断错了全盘乱 |

---

### 2.9 Agentic Coding — "AI 程序员"

> **原理**：让 AI Agent 自主完成软件开发的完整流程——理解需求、规划任务、读写文件、执行命令、运行测试、修复错误，实现从"代码补全"到"自主编程"的范式跃迁。

| 属性 | 内容 |
|------|------|
| **名号** | AI 程序员 |
| **一句话记忆** | "从补全代码到自己写代码" |
| **生动类比** | 像招了一个初级程序员，你给需求他写代码跑测试改bug，你只管Review |
| **招式特点** | 自主完成编码-测试-调试闭环，代表产品 Devin/Cursor/Claude Code |
| **最佳战场** | 功能开发、Bug 修复、代码重构 |
| **致命弱点** | 长程任务易跑偏，对复杂架构理解有限 |

### 2.10 Long-running Agents (长时运行 Agent) — "持久战"

> **原理**：能够持续运行数小时到数周的 Agent 系统，核心解决上下文管理（Compaction）、状态持久化（Checkpointing）、错误恢复（Resumability）和目标保持（Goal Maintenance）四大挑战。

| 属性 | 说明 |
|------|------|
| **门派** | 将帅门（自主规划） |
| **内力等级** | ⭐⭐⭐⭐⭐ |
| **招式特点** | Checkpoint 持久化+Compaction 压缩+人在回路+预算控制 |
| **适用场景** | 数据分析、持续监控、长期项目协作、自动化运维 |
| **致命弱点** | 工程复杂度极高；长任务中不可预测的失败模式多 |
| **代表实现** | Devin、OpenClaw、Claude Code |

---

### 2.11 Durable Execution (持久化执行引擎) — "不死之身"

> **原理**：保证 Agent 工作流"永不丢失"的执行范式。每个步骤状态持久化到外部存储，进程崩溃后自动从最后检查点恢复。代表实现：Temporal、Inngest、Restate、DBOS。

| 属性 | 说明 |
|------|------|
| **门派** | 将帅门（自主规划） |
| **内力等级** | ⭐⭐⭐⭐⭐ |
| **招式特点** | 状态持久化+幂等重试+崩溃恢复+人在回路 |
| **适用场景** | 生产级 Agent、Long-running Agent、企业工作流 |
| **致命弱点** | 增加系统复杂度；持久化开销影响延迟 |
| **代表实现** | Temporal、Inngest、Restate、DBOS |

## 🔍 侦探门 — 检索增强类

> **门派心法**：耳听为虚，查证为实。侦探们让 Agent 能查到外部知识、还能查得准。
> **完整武学**：[03_检索增强类.md](03_检索增强类.md)

### 3.1 Standard RAG (Retrieval-Augmented Generation，检索增强生成) — "查字典"

> **原理**：将外部知识切块向量化存储，查询时检索相关片段拼入提示词，让 LLM 基于检索到的上下文生成回答，解决模型知识不足和幻觉问题。

| 属性 | 内容 |
|------|------|
| **名号** | 查字典 |
| **一句话记忆** | "遇到不懂的，翻参考书" |
| **生动类比** | 像开卷考试，遇到问题就翻书找答案 |
| **招式特点** | 把外部知识切块向量化，检索相关片段拼进 prompt 再生成回答 |
| **最佳战场** | 知识密集问答、私有知识库问答 |
| **致命弱点** | 检索不准就答错，"翻错页"比"不翻书"更误导 |

---

### 3.2 Self-RAG — "自省查证"

> **原理**：模型自主决定是否检索、检索结果是否相关、回答是否有支撑，通过 Reflection Token 实现检索-生成-验证的自适应闭环。

| 属性 | 内容 |
|------|------|
| **名号** | 自省查证 |
| **一句话记忆** | "查完还要问自己：这靠谱吗？" |
| **生动类比** | 像查到资料后还要判断"这个来源可信吗" |
| **招式特点** | 模型自己决定要不要检索、检索结果要不要用、最终回答要不要支持 |
| **最佳战场** | 需要高可靠性的问答、易产生幻觉的场景 |
| **致命弱点** | 自我判断也可能误判，"自以为靠谱其实不靠谱" |

---

### 3.3 Corrective RAG — "纠错侦探"

> **原理**：检索后用评估器给文档打分，可信的直接用，不可信的转 Web 搜索纠正，通过"检索+纠错"双保险提升答案准确性。

| 属性 | 内容 |
|------|------|
| **名号** | 纠错侦探 |
| **一句话记忆** | "查到线索？先验真伪" |
| **生动类比** | 像侦探查到线索要去验证，不能轻信 |
| **招式特点** | 检索后用评估器给文档打分，可信的用、不可信的转 web 搜索纠正 |
| **最佳战场** | 知识库质量参差不齐、需要高准确率的场景 |
| **致命弱点** | 评估器判断失误会"误杀"好文档或"放行"坏文档 |

---

### 3.4 RAISE — "对话侦探"

> **原理**：将对话记忆与检索结合，在多轮对话中动态检索相关知识，解决长对话中上下文丢失和信息遗忘问题。

| 属性 | 内容 |
|------|------|
| **名号** | 对话侦探 |
| **一句话记忆** | "边聊天边查资料" |
| **生动类比** | 像客服边和你聊边查知识库 |
| **招式特点** | 把对话记忆和检索结合，在多轮对话中动态检索相关知识 |
| **最佳战场** | 多轮对话、客服、需要结合上下文检索的场景 |
| **致命弱点** | 对话越长记忆管理越乱，检索时机难把握 |

---

### 3.5 Active RAG — "主动出击"

> **原理**：模型判断当前信息是否充足，不够时主动发起多轮检索补充，通过"按需追问"策略提升复杂问题的信息覆盖度。

| 属性 | 内容 |
|------|------|
| **名号** | 主动出击 |
| **一句话记忆** | "信息不够？我主动追问" |
| **生动类比** | 像记者采访觉得信息不够会追问 |
| **招式特点** | 模型判断当前信息是否充足，不够就主动发起多轮检索补充 |
| **最佳战场** | 复杂问题、单次检索不够用的深度问答 |
| **致命弱点** | "追问"刹不住车会无限检索，token 失控 |

---

### 3.6 GraphRAG — "关系网络"

> **原理**：将知识构建为实体关系图，检索时既查实体又查关系，支持需要全局性推理和跨实体关联的复杂问答。

| 属性 | 内容 |
|------|------|
| **名号** | 关系网络 |
| **一句话记忆** | "不只查文档，还查关系" |
| **生动类比** | 像画人物关系图来破案，不只看单个人物档案 |
| **招式特点** | 把知识构建成实体关系图，检索时既查实体又查关系，支持全局性问题 |
| **最佳战场** | 关系推理、全局总结、多实体关联分析 |
| **致命弱点** | 建图成本高，图维护和查询都复杂 |

---

### 3.7 HyDE (Hypothetical Document Embeddings，假设文档嵌入) — "假想敌"

> **原理**：先让 LLM 生成一个"假想答案"文档，再用它去检索相似的真实文档，通过缩小"问题与答案的表述差异"提升检索召回率。

| 属性 | 内容 |
|------|------|
| **名号** | 假想敌 |
| **一句话记忆** | "先假设答案长啥样，再去找" |
| **生动类比** | 像先画个嫌疑人画像再去匹配 |
| **招式特点** | 先让 LLM 生成一个"假想答案"文档，再用它去检索相似的真实文档 |
| **最佳战场** | 问题与答案表述差异大的检索场景、抽象查询 |
| **致命弱点** | 假想答案跑偏了，检索方向就全错 |

---

### 3.8 Self-Ask — "拆问专家"

> **原理**：将复杂问题拆成一系列子问题，逐个检索答案再组合，通过"大问题拆小、逐个击破"实现多跳推理。

| 属性 | 内容 |
|------|------|
| **名号** | 拆问专家 |
| **一句话记忆** | "大问题拆小问题，逐个击破" |
| **生动类比** | 像"中国GDP是多少"拆成"中国是什么"+"GDP怎么算" |
| **招式特点** | 把复杂问题拆成一系列子问题，逐个检索答案再组合 |
| **最佳战场** | 多跳问答、需要组合多方面信息的问题 |
| **致命弱点** | 拆问方式不当会越拆越乱，子问题之间衔接不上 |

---

### 3.9 FLARE (Forward-Looking Active Retrieval，前瞻式主动检索) — "边走边看"

> **原理**：边生成边监测置信度（通过 token 概率），发现不确定就暂停检索再继续，实现"边写边查"的前瞻式主动检索。

| 属性 | 内容 |
|------|------|
| **名号** | 边走边看 |
| **一句话记忆** | "写到一半发现不懂，停下来查" |
| **生动类比** | 像写论文写到一半发现引用不会，停下来查文献 |
| **招式特点** | 边生成边监测置信度，发现不确定就暂停、检索、再继续 |
| **最佳战场** | 长文生成、需要边写边查证的任务 |
| **致命弱点** | 频繁停顿查证拖慢速度，置信度判断不一定准 |

---

### 3.10 Speculative RAG — "草稿快枪手"

> **原理**：双模型架构：小模型并行生成多个草稿，大模型验证并合成最终答案，兼顾速度与质量。

| 属性 | 内容 |
|------|------|
| **名号** | 草稿快枪手 |
| **一句话记忆** | "小弟先写草稿，大哥来把关" |
| **生动类比** | 像实习生快速写初稿，主编审核定稿，又快又稳 |
| **招式特点** | 双模型架构：小模型并行生成草稿，大模型验证合成，兼顾速度与质量 |
| **最佳战场** | 实时问答、低延迟客服、需要兼顾速度和质量的场景 |
| **致命弱点** | 双模型架构增加系统复杂度，小模型草稿太差时大模型验证反而更慢 |

---

### 3.11 Adaptive RAG — "因材施教"

> **原理**：先评估查询复杂度，再选择 no retrieval / single-step / multi-step 检索策略，按需分配检索资源。

| 属性 | 内容 |
|------|------|
| **名号** | 因材施教 |
| **一句话记忆** | "简单问题不查书，中等问题查一次，复杂问题查多次" |
| **生动类比** | 像学霸根据题目难度决定翻书次数，简单题直接答，难题多查几页 |
| **招式特点** | 先评估查询复杂度，再选择 no retrieval / single-step / multi-step 检索策略 |
| **最佳战场** | 查询复杂度差异大的混合场景，兼顾效率与质量 |
| **致命弱点** | 复杂度分类器判断错误会导致策略选择失当 |

---

### 3.12 LightRAG — "轻功"

> **原理**：GraphRAG 的轻量化替代，通过"双层检索"（实体级+关系级）和增量更新机制，在保持图谱推理能力的同时大幅降低构建和查询成本。

| 属性 | 内容 |
|------|------|
| **名号** | 轻功 |
| **一句话记忆** | "图谱推理不用那么重" |
| **生动类比** | GraphRAG 是全量测绘地图，LightRAG 是只标关键地标+道路的简图，够用还快 |
| **招式特点** | 双层检索(low-level+high-level) + 增量更新，比 GraphRAG 轻量 |
| **最佳战场** | 中小规模知识库、需要图谱推理但预算有限 |
| **致命弱点** | 图谱质量不如 GraphRAG 全面 |

---

### 3.13 Agentic RAG (代理式 RAG) — "智能调查员"

> **原理**：第三代 RAG 架构，将 Agent 的规划、反思、工具使用能力注入 RAG 流程。支持多跳推理、跨源检索、自我检查和失败重检索。

| 属性 | 说明 |
|------|------|
| **门派** | 侦探门（检索增强） |
| **内力等级** | ⭐⭐⭐⭐⭐ |
| **招式特点** | 查询重写+多跳检索+工具路由+自我检查+失败重检索 |
| **适用场景** | 多跳推理、跨源检索、合规审查 |
| **致命弱点** | 延迟高（3-8 LLM 调用+2-6 次检索）；可能过度检索 |
| **代表实现** | LangGraph、LlamaIndex Agentic RAG |

---

### 3.14 CAG (Cache-Augmented Generation，缓存增强生成) — "知识缓存"

> **原理**：将整个知识库预加载到 LLM 的长上下文窗口（如 Gemini 2M tokens）中，通过 KV Cache 缓存，无需运行时检索。首次加载后后续查询直接从缓存读取，延迟极低。

| 属性 | 说明 |
|------|------|
| **门派** | 侦探门（检索增强） |
| **内力等级** | ⭐⭐⭐ |
| **招式特点** | 全量预加载+KV Cache+无运行时检索 |
| **适用场景** | 静态知识库、FAQ、产品文档 |
| **致命弱点** | 知识库大小受上下文窗口限制；更新成本高 |
| **代表实现** | Gemini 2.5 Pro Long Context |

---

### 3.15 Multimodal RAG (多模态 RAG) — "跨模态侦探"

> **原理**：解决企业 RAG 的 Modality Gap（模态鸿沟）问题——结构化（SQL）+ 非结构化（文档）数据融合检索。通过统一嵌入空间或层级多 Agent 拓扑实现跨模态推理。

| 属性 | 说明 |
|------|------|
| **门派** | 侦探门（检索增强） |
| **内力等级** | ⭐⭐⭐⭐ |
| **招式特点** | 跨模态检索+SQL+文档融合+supervisor-worker 拓扑 |
| **适用场景** | 企业数据分析、金融研究、合规审查 |
| **致命弱点** | 工程复杂度高；跨模态对齐困难 |
| **代表实现** | Protocol-H、ColPali |

---

### 3.16 Reranking (重排序) — "精排师"

> **原理**：RAG 工程化中收益最高的单一优化手段。"先粗排后精排"——Bi-Encoder 快速检索 Top-50 候选，Cross-Encoder 精排到 Top-5。可提升正确率 18-31%，仅增加 50-200ms 延迟。

| 属性 | 说明 |
|------|------|
| **门派** | 侦探门（检索增强） |
| **内力等级** | ⭐⭐⭐⭐ |
| **招式特点** | 两阶段检索（Bi-Encoder 粗排 + Cross-Encoder 精排） |
| **适用场景** | 企业 RAG 系统、知识密集型问答、精确信息检索 |
| **致命弱点** | Cross-Encoder 增加延迟；对 Top-K 设置敏感 |
| **代表实现** | bge-reranker、Cohere Rerank、Jina Reranker |

## 🤝 联盟门 — 多智能体协作类

> **门派心法**：独木不成林。联盟门让多个 Agent 组队，各展所长协同完成任务。

### 4.1 AutoGen — "圆桌会议"

## 📚 记忆门 — 记忆与状态管理类

> **门派心法**：前事不忘，后事之师。记忆门让 Agent 能记住该记的、忘掉该忘的。
> **完整武学**：[05_记忆与状态管理类.md](05_记忆与状态管理类.md)

### 5.1 MemGPT — "记忆管家"

> **原理**：借鉴操作系统分级内存思想，将记忆分为主存（上下文窗口）和外存（持久化存储），自动管理记忆的换入换出，突破上下文窗口限制。

| 属性 | 内容 |
|------|------|
| **名号** | 记忆管家 |
| **一句话记忆** | "有个秘书帮你管记忆" |
| **生动类比** | 像有个秘书帮你记重要的事，忘掉不重要的 |
| **招式特点** | 借鉴操作系统分级内存思想，分主存/外存，自动管理记忆换入换出 |
| **最佳战场** | 长对话、需要长期记忆的助手、上下文超长的场景 |
| **致命弱点** | 换入换出有开销，关键记忆被"换出去"会断片 |

---

### 5.2 Generative Agents — "日记达人"

> **原理**：Agent 记录日常经历形成记忆流，按近期性/重要性/相关性三维检索，定期反思总结形成高层洞察指导后续行为。

| 属性 | 内容 |
|------|------|
| **名号** | 日记达人 |
| **一句话记忆** | "写日记+定期复盘" |
| **生动类比** | 像每天写日记，定期翻看总结人生经验 |
| **招式特点** | Agent 记录日常经历，定期反思总结形成高层洞察，指导后续行为 |
| **最佳战场** | 虚拟小镇、模拟社会、需要"人格连续性"的 Agent |
| **致命弱点** | 日记写多了占地方，反思不及时记忆就"过期" |

---

### 5.3 Reflection-based Memory — "复盘高手"

> **原理**：对过往交互/任务进行反思，提炼成抽象经验存入记忆供后续调用，将"经历"转化为"教训"。

| 属性 | 内容 |
|------|------|
| **名号** | 复盘高手 |
| **一句话记忆** | "经历不总结等于白经历" |
| **生动类比** | 像每次项目结束后写复盘报告 |
| **招式特点** | 对过往交互/任务进行反思，提炼成经验存入记忆供后续调用 |
| **最佳战场** | 需要持续改进的 Agent、经验积累型任务 |
| **致命弱点** | 复盘错了会把"错误经验"固化下来 |

---

### 5.4 RAG-based Memory — "索引大师"

> **原理**：将历史记忆向量化存储，需要时按语义检索相关记忆召回，实现"按需回忆"的大规模记忆管理。

| 属性 | 内容 |
|------|------|
| **名号** | 索引大师 |
| **一句话记忆** | "给记忆建索引，随时查" |
| **生动类比** | 像给日记本建目录，需要时按关键词查 |
| **招式特点** | 把历史记忆向量化存储，需要时检索相关记忆召回 |
| **最佳战场** | 大量历史交互、需要按需召回记忆的长程任务 |
| **致命弱点** | 索引/检索不准，"想不起来的还是想不起来" |

---

### 5.5 Episodic Memory — "情景日记"

> **原理**：记录具体事件（时间、地点、人物、经过），需要时回忆相似情景，通过"情景类比"指导当前决策。

| 属性 | 内容 |
|------|------|
| **名号** | 情景日记 |
| **一句话记忆** | "记具体事件，不记抽象教训" |
| **生动类比** | 像写日记记"今天发生了什么"，而非"我学到了什么" |
| **招式特点** | 记录具体事件（时间、地点、人物、经过），需要时回忆相似情景 |
| **最佳战场** | 需要情景类比的任务、案例推理、个性化交互 |
| **致命弱点** | 记了一堆细节却不会"举一反三"，相似情景匹配不上 |

---

### 5.6 Letta — "不眠管家"

> **原理**：MemGPT 的 2024 演进版，支持状态持久化、记忆工具调用和 sleep-time compute（离线记忆整理），实现跨会话的有状态 Agent。

| 属性 | 内容 |
|------|------|
| **名号** | 不眠管家 |
| **一句话记忆** | "白天陪你聊天，晚上整理记忆" |
| **生动类比** | 像管家白天服务客人，晚上整理笔记、归纳总结，第二天服务更到位 |
| **招式特点** | MemGPT的2024演进版，支持状态持久化、记忆工具调用、sleep-time compute（离线记忆整理） |
| **最佳战场** | 长期陪伴型Agent、需要跨会话记忆的有状态应用 |
| **致命弱点** | 状态管理复杂度高，sleep-time compute增加离线成本 |

---

### 5.7 Mem0 — "记忆管家"

> **原理**：通过"提取-存储-检索-更新"四步管理用户级长期记忆。与 MemGPT 的"操作系统式"分级记忆不同，Mem0 采用更轻量的"记忆即数据库"思路。

| 属性 | 内容 |

### 5.8 Dreaming (梦境记忆整合) — "AI 做梦"

> **原理**：Agent 在空闲时异步读取全部对话历史，自动合成统一的"记忆状态"，核心创新是"时间自我修正"——记忆会自动感知时间推移并更新状态（如"要去新加坡"→"去了新加坡"），无需用户干预。

| 属性 | 说明 |
|------|------|
| **门派** | 记忆门（记忆管理） |
| **内力等级** | ⭐⭐⭐⭐ |
| **招式特点** | 全量历史合成+时间自我修正+推理时上下文注入 |
| **适用场景** | 长期个性化助手、跨年对话记忆、用户画像自动维护 |
| **致命弱点** | 需要全量历史访问权限，隐私敏感；后台计算成本高 |
| **代表实现** | OpenAI ChatGPT Dreaming V3（2026.06） |

| **最佳战场** | 长程对话关系推理、用户画像演化、世界模型记忆 |
| **致命弱点** | 三元组抽取易错；图越大遍历越慢，需剪枝和索引 |

## 🔧 工匠门 — 框架与编程范式类

> **门派心法**：工欲善其事，必先利其器。工匠门从系统层面组织 LLM，是江湖的基建能手。
> **完整武学**：[06_框架与编程范式类.md](06_框架与编程范式类.md)

### 6.1 DSPy (Declarative Self-improving Python，声明式自改进Python) — "自动调参"

> **原理**：将 prompt 优化从手工调参变为可编程、可编译的自动化过程，通过声明式签名+模块化组合自动搜索最优 prompt。

| 属性 | 内容 |
|------|------|
| **名号** | 自动调参 |
| **一句话记忆** | "别手写prompt了，让程序自己调" |
| **生动类比** | 像自动调焦相机，不用手动调 |
| **招式特点** | 把 prompt 优化变成可编程、可编译的过程，自动搜索最优 prompt 和权重 |
| **最佳战场** | 需要系统化优化 prompt 的场景、可复现的 LLM 工程 |
| **致命弱点** | 学习曲线陡，小项目上手成本高 |

---

### 6.2 Flow Engineering — "流程图编程"

> **原理**：用图结构编排 LLM 调用，支持循环、分支、回溯等复杂控制流，将 LLM 工程从线性管道升级为可迭代的工作流图。

| 属性 | 内容 |
|------|------|
| **名号** | 流程图编程 |
| **一句话记忆** | "画个流程图就能编程" |
| **生动类比** | 像画流程图定义工作流，节点间可循环可分支 |
| **招式特点** | 用图结构编排 LLM 调用，支持循环、分支、回溯等复杂控制流 |
| **最佳战场** | 复杂工作流、需要迭代精炼的流程 |
| **致命弱点** | 流程图画复杂了自己都看不懂，调试困难 |

---

### 6.3 Map-Reduce — "分而治之"

> **原理**：将大任务拆成子任务并行处理（Map），再聚合结果（Reduce），通过分而治之策略加速大批量数据处理。

| 属性 | 内容 |
|------|------|
| **名号** | 分而治之 |
| **一句话记忆** | "大任务拆小，并行做，再合并" |
| **生动类比** | 像搬砖：10个人分头搬，最后堆一起 |
| **招式特点** | 把大任务拆成子任务并行处理（Map），再聚合结果（Reduce） |
| **最佳战场** | 大批量数据处理、长文档摘要、可并行任务 |
| **致命弱点** | 子任务之间有依赖时并行不了，合并阶段易成瓶颈 |

---

### 6.4 Router/MoE (Mixture of Experts，混合专家模型) — "前台调度"

> **原理**：用路由器把请求分发到最合适的专家模型/子任务处理器，通过"按需分配"平衡质量与成本。

| 属性 | 内容 |
|------|------|
| **名号** | 前台调度 |
| **一句话记忆** | "看人下菜碟，按需分配" |
| **生动类比** | 像医院导诊台，根据症状分到不同科室 |
| **招式特点** | 用路由器把请求分发到最合适的专家模型/子任务处理器 |
| **最佳战场** | 多模型协同、按任务类型分流、成本优化 |
| **致命弱点** | 路由分错了就送错"科室"，越治越糟 |

---

### 6.5 Structured Output — "格式大师"

> **原理**：用 JSON Schema/函数签名约束 LLM 输出格式，保证生成结果可被程序直接解析消费。

| **最佳战场** | 函数调用、数据抽取、需要程序消费 LLM 输出的场景 |
| **致命弱点** | 复杂 schema 模型容易"填错格"，强约束下内容质量可能下降 |

| **致命弱点** | 协议虽标准，工具实现质量参差不齐 |

### 6.7 Semantic Kernel — "插件大师"

### 6.8 Prompt Chaining — "流水线"

> **原理**：把任务拆成多个 prompt 串联，前一步输出作为后一步输入，每步可加校验门控，实现流水线式逐步质检。

| 属性 | 内容 |

| **招式特点** | 把任务拆成多个 prompt 串联，前一步输出作为后一步输入，可加校验 |
| **最佳战场** | 多步处理流程、需要逐步质检的任务 |
| **致命弱点** | 链条越长越脆，一环出错全链崩 |

### 6.9 Orchestrator-Workers — "总调度"

> **原理**：一个 Orchestrator Agent 动态分解任务并分派给多个 Worker Agent 并行执行，最后汇总结果。与 Map-Reduce 的区别是 LLM 动态决定如何拆分。

| 属性 | 内容 |
|------|------|
| **名号** | 总调度 |
| **一句话记忆** | "我来分活，你们干，我来合" |
| **生动类比** | 项目经理把大项目拆成子任务分给组员并行干，最后汇总成报告 |
| **招式特点** | Orchestrator 动态分解 → Workers 并行 → Orchestrator 汇总 |
| **最佳战场** | 需要动态分解的复杂任务、多来源信息汇总 |
| **致命弱点** | Orchestrator 分解质量决定整体效果 |

### 6.10 Evaluator-Optimizer — "打磨师"

> **原理**：Generator 生成输出后，由独立 Evaluator 评估质量并给出反馈，Generator 根据反馈优化，循环到达标。与 Self-Refine 区别是使用独立评估角色。

| 属性 | 内容 |
|------|------|
| **名号** | 打磨师 |
| **一句话记忆** | "写完让另一个人审，审完改，改完再审" |
| **生动类比** | 像论文写作：作者写初稿，审稿人提意见，作者修改，循环到接收 |
| **招式特点** | Generator → Evaluator → 反馈 → 优化，循环到达标 |
| **最佳战场** | 翻译润色、文案优化、代码审查 |
| **致命弱点** | 循环次数多成本高，可能过度优化 |

---

### 6.11 Context Engineering / Compaction (上下文工程) — "上下文炼金术"

> **原理**：从"写好一次性提示词"升级为"管理 Agent 整个生命周期的上下文"。核心包括上下文压缩（Compaction）、上下文衰减防御（Context Rot 检测）和按需加载（JIT Loading）。2026 年成为 Agent 工程化的核心战场。

| 属性 | 说明 |
|------|------|
| **门派** | 工匠门（框架范式） |
| **内力等级** | ⭐⭐⭐⭐ |
| **招式特点** | Compaction 压缩+Context Rot 检测+JIT 加载 |
| **适用场景** | 长任务 Agent、多轮对话系统、工具密集型 Agent |
| **致命弱点** | 压缩可能丢失关键信息；Context Rot 检测增加额外 LLM 调用 |
| **代表实现** | Claude Code、Cursor、Claude Agent SDK |

### 6.12 Subagents (子代理模式) — "分身术"

> **原理**：Anthropic 在 Claude Code 中产品化的多 Agent 协作模式。主 Agent 通过 Task 工具调度子 Agent，每个子 Agent 拥有独立上下文窗口和专用工具集。核心优势是"上下文隔离"，是生产环境中最稳定的多 Agent 模式。

| 属性 | 说明 |
|------|------|
| **门派** | 工匠门（框架范式） |
| **内力等级** | ⭐⭐⭐⭐⭐ |
| **招式特点** | 上下文隔离+专用工具集+摘要返回+并行执行 |
| **适用场景** | 代码开发（Claude Code）、复杂研究、多步骤任务 |
| **致命弱点** | 子 Agent 间无法直接通信；摘要可能丢失关键信息 |
| **代表实现** | Claude Code（2025.02）、Claude Agent SDK（2025.09） |

---

### 6.13 Meta-Controller (元控制器) — "调度总管"

> **原理**：用统一的评分函数给候选子 Agent 打分（任务匹配度+空闲度+健康度），按优先级动态分配任务，负载感知全局路由。

| 属性 | 内容 |
|------|------|
| **名号** | 调度总管 |
| **一句话记忆** | "谁更合适、谁更闲，就派谁" |
| **生动类比** | 像大酒店值班经理看名单选人接待客人：既看专长、也看是否在忙、还看近期状态 |
| **招式特点** | 语义匹配+负载感知+健康度评分，全局调度而非静态路由 |
| **最佳战场** | 多模型 Agent 集群、长时任务编排、需要故障转移的生产环境 |
| **致命弱点** | 评分函数设计门槛高；评分延迟会拖累调度本身 |

## 🔄 修炼门 — 自我纠错与迭代改进类

> **原理**：任务失败后生成口头反思存入记忆，下次尝试时参考避免重蹈覆辙，通过"失败→反思→改进"的循环实现从错误中学习。

| **最佳战场** | 多轮试错任务、需要从失败中学习的场景 |
| **致命弱点** | 反思错了会"记错教训"，多轮试错成本高 |

### 9.4 Iterative Refinement — "千锤百炼"

> **原理**：对输出进行多轮迭代精炼，每轮基于反馈持续改进，通过"千锤百炼"式的打磨逼近最优结果。

| **致命弱点** | "招式"组装错了，后面全白搭 |

### 9.6 Meta-Agent / Self-Improving Agent (自我进化元 Agent) — "造物主"

| **生动类比** | 像把常用文件放桌面，不用每次都翻柜子 |
| **招式特点** | 缓存高频查询结果，命中直接返回，省时省钱 |
| **最佳战场** | 高频重复查询、相似问题密集的场景 |
| **致命弱点** | 缓存失效不及时会"返旧货"，缓存管理也有成本 |

---

### 12.4 Model Routing — "因材施教"

> **原理**：按任务难度/类型动态路由到不同规格模型，简单题用小模型、难题用大模型，通过"因材施教"平衡质量与成本。

| 属性 | 内容 |
|------|------|
| **名号** | 因材施教 |
| **一句话记忆** | "简单题用小模型，难题用大模型" |
| **生动类比** | 像因材施教，小学题不用请教授 |
| **招式特点** | 按任务难度/类型动态路由到不同规格模型，平衡质量与成本 |
| **最佳战场** | 大规模调用、成本敏感场景、任务难度差异大的应用 |
| **致命弱点** | 难度判断错了，"难题给小学生做"或"简单题请教授" |

---

### 12.5 Prompt Caching & Batch API — "前缀管家"

> **原理**：提供商侧缓存长 prompt 的 KV cache 跳过 prefill（成本降50%）；Batch API 异步处理享50%折扣，双管齐下降低 API 成本。

| 属性 | 内容 |
|------|------|
| **名号** | 前缀管家 |
| **一句话记忆** | "同样的开场白不用重复背，批量办事还能打折" |
| **生动类比** | 像管家把你的常用台词背好了，下次直接从中间开始说；批量办事还能享受批发价 |
| **招式特点** | 提供商侧缓存长prompt的KV cache (Key-Value cache，键值缓存)跳过prefill（成本降50%）；Batch API异步处理享50%折扣 |
| **最佳战场** | 长system prompt应用、多轮对话、离线批量处理 |
| **致命弱点** | 缓存命中需前缀完全一致，Batch API有24h延迟 |

---

### 12.6 Speculative Decoding — "草稿快进"

> **原理**：用小模型快速生成候选 token，再用大模型并行验证。验证通过的多 token 一次性接受，实现 2-3 倍推理加速且不损失质量。

| 属性 | 内容 |
|------|------|
| **名号** | 草稿快进 |
| **一句话记忆** | "小弟先写草稿，大哥扫一眼通过" |
| **生动类比** | 像领导让实习生先起草文件，自己只做审批，通过就直接用，不通过重写 |
| **招式特点** | 小模型草拟 + 大模型并行验证，2-3x 加速，质量完全一致 |
| **最佳战场** | 自托管模型推理、Agent 多轮调用、高并发场景 |
| **致命弱点** | 需要 GPU 自托管，对短输出加速有限 |

---

### 12.7 Reflexive Agent (反射型 Agent) — "反射弧"

> **原理**：在"内部推理"和"外部工具调用"两条路径间动态路由。先评估问题难度，简单问题直接内部推理，复杂或时效性问题自动升级到工具调用，置信度不足时自动回退再升级。

| 属性 | 内容 |
|------|------|
| **名号** | 反射弧 |
| **一句话记忆** | "想得通就别动手，想不通再上工具" |
| **生动类比** | 像老司机开车，熟路就脑内决策，遇到陌生路就开导航查实时路况 |
| **招式特点** | 难度评估+置信度检查+自动升级，路由在"行动模式"层而非"模型"层 |
| **最佳战场** | 成本敏感的多任务场景、需要按需降本又不失准的 Agent |
| **致命弱点** | 难度评估本身可能误判；置信度阈值难调，容易"该升级没升级" |
