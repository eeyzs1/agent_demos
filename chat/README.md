# agent_demos


## Date
你需要用做一个云原生应用，由K8S进行编排，后端用python版本的fastapi+pydantic搭建，需要用Redis 做缓存 + Kafka 做消息队列。前端平台主要为安卓和web端，使用flutter开发。
这个应用功能包括：
1.用户注册和登录，这些信息需要使用python orm保存到mysql数据库中。将用户池分为四类，并在数据库中标记：
🟢 实时在线 (Real-time)：心跳正常，5 分钟内有过操作。
🟡 活跃 (Active)：24 小时内登录过。
🟠 沉睡 (Sleeping)：7-30 天未登录，但账号未注销。
⚫ 僵尸 (Zombie)：>30 天未登录，或推送长期失败，或明确卸载。
2.根据我上传的“情侣配对软件新用户分析问题框架”，按照其中的六项，用python版本的LangGraph+deepagents构建一个对话agent，这个agent可以通过不同的模型，system prompt，skills设置大模型成为对应领域的专家，来模拟各种情景来实现对用户的六个维度的信息获取，按照用户在不同情景下做出的选择来生成这个用户的六维信息向量，根据大模型的response，前端需要相应的出现选项或者问答题，之后的结果回传到后端agent，agent将其作为上下文的一部分继续与大模型对话，进行多轮对话后，由agent要求大模型进行一个最终的评判，并将大模型的返回结果向量化。对话内容则先存在内存里，因为是从六个维度进行分析，所以一个用户会有六个向量，agent在集齐这六个向量后，需要合成一个代表这个用户特质的特质向量。用户的对话内容则进行合并后，发给设定成总结大师的大模型进行总结，之后总结内容像是文档一样被存储，被存总结的元数据和特质向量一起存入向量数据库。
3.当用户完成问卷后，当他在前端点击匹配时，并选择配对人数（例如3）时，后端接收到这个信息后，可以按照他的特质向量从向量数据库中来选择适配的其他用户的特质向量，然后选出前若干名其他匹配到的用户作为候选人，然后按照活跃度进行二次筛选，最终根据用户之前选择的人数来确定最终的候选人群，然后按照匹配到的元数据，找到之前的对话总结，将候选人信息和对话总结发给前端，前端会渲染出这些信息给用户，总结会被渲染成一部分信息，由用户点开后展示详细信息，用户可以从这些候选人中选择一名进行对话
4.用户选好后，可以通过前端的app发送消息给被选者，后端需要维护这两人之间的对话，模仿微信的通讯模式，但是显示是否送达和已阅读。


## Chat
# Role
你是一位资深的全栈架构师和即时通讯（IM）系统专家，精通WebSocket、高并发处理及移动端开发。

# Goal
请帮我设计并生成一个最小可行性产品（MVP）的代码框架。这是一个独立的即时通讯App，界面风格类似微信，核心功能仅包含单聊和群聊，但必须严格实现以下两个状态机制：
1. 消息送达（Delivered）：消息成功推送到对方设备。
2. 消息已读（Read）：对方打开对话框查看消息后，发送方能看到“已读”标记。

# Requirements
1. **技术栈建议**：
   - 前端：Flutter 或 React Native (跨平台)。
   - 后端：Node.js (Socket.io) 或 Go (Gin + WebSocket)。
   - 数据库：MongoDB (存储消息) + Redis (处理在线状态和临时回执)。
2. **核心逻辑实现**：
   - 请详细解释如何设计协议来实现“已读回执”。例如：当接收端渲染消息列表时，如何触发ACK信号回传给发送端？
   - 数据库Schema设计：消息表需要包含哪些字段来区分 `sent`, `delivered`, `read` 状态及时间戳？
3. **代码生成**：
   - 提供后端WebSocket处理消息发送和回执的核心代码片段。
   - 提供前端监听回执并更新UI（如显示双勾图标）的逻辑代码。
4. **安全性**：简要说明如何防止伪造“已读”状态。

# Output Format
请先输出架构图解文字版，然后分模块提供关键代码，最后给出部署步骤。
